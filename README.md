# AELITA PRODUCTION — сайт

Статический сайт театральной продюсерской компании AELITA PRODUCTION.
Домен: **aelita-production.ru** (GitHub Pages, см. `CNAME`).

Чистый HTML/CSS/JS без сборки — что лежит в репозитории, то и деплоится.

## Структура

```
/
├── index.html              главная
├── <раздел>/index.html     каждая страница — своя папка + index.html
│                           (даёт чистые адреса вида /porfiriy, /tickets и т.д.)
├── assets/style.css        общие стили (см. ниже)
├── icons/, favicon.ico     иконки, manifest.json — PWA
├── sw.js, manifest.json    service worker и манифест для «Установить как приложение»
├── robots.txt, sitemap.xml
└── 404.html, offline.html  служебные страницы
```

## ⚠️ CSS: как устроено и что нельзя ломать

Стили лежат в двух местах:

1. **`assets/style.css`** — общее для всего сайта: шапка, мобильное меню, футер,
   кнопки, карточки событий, брейкпоинты. Подключается через
   `<link rel="stylesheet" href="/assets/style.css">`.
2. **`<style>` внутри каждой страницы** — то, что уникально именно для неё
   (свои отступы hero-блока, свой `--maxw`, специфичные компоненты).

**Три страницы НЕ подключают `assets/style.css` вообще** и должны быть
полностью самодостаточны — весь нужный CSS у них внутри `<style>`:

- `404.html`
- `offline.html`
- `app/index.html`

Это осознанно: страница 404 и офлайн-заглушка обязаны отрисоваться, даже
если с сайтом или сетью что-то не так. Если добавляешь этим трём страницам
новый визуальный элемент — стили для него нужно писать **прямо в их
собственном `<style>`**, не полагаясь на общий файл.

### Главное правило при правке общих стилей

Внешний `assets/style.css` в каскаде всегда идёт **раньше** локального
`<style>` страницы. Значит:

- Если у страницы есть **своё** базовое правило для селектора (например,
  свой `.sec-head{margin-bottom:...}`), а в `assets/style.css` для этого же
  селектора добавить `@media(...)`-переопределение — оно **не сработает**
  на этой странице, потому что локальное правило (без медиа-условия) всегда
  идёт позже и просто перебивает медиа-запрос.
- Если правишь что-то в `assets/style.css`, **проверяй, нет ли у страниц
  своего локального правила с тем же селектором** — если есть, адаптивное
  переопределение нужно продублировать и в `<style>` этой страницы тоже.
- В `assets/style.css` не должно быть **двух голых (без `@media`) правил
  для одного и того же селектора** — выигрывает то, что физически ниже по
  файлу, и это легко случайно поломать одной группе страниц в пользу другой.

Если что-то в вёрстке выглядит иначе на разных страницах непонятно почему —
в 9 случаях из 10 причина именно в этом.

### `:root{--maxw}`

Ширина контейнера контента — переменная, разная для двух групп страниц:
- `1320px` — все обычные страницы (задано в `assets/style.css` по умолчанию)
- `900px` — юридические страницы (`oferta`, `privacy`, `cookies`,
  `documents`, `bot`), заданы локально в каждой из них

## Юридические страницы и индексация

`robots.txt` закрывает от индексации `/oferta`, `/privacy`, `/cookies` —
это осознанно, не баг. При добавлении новой страницы не забудь внести её в
`sitemap.xml`.

нужно будет самостоятельно захостить `.woff2` локально.

## `.page-hero` — один класс, два разных смысла

На части страниц `.page-hero` — простой текстовый блок (заголовок + подпись,
статичный фон, обычный `padding`). На страницах с видео на весь экран
(`porfiriy`, `iov`, `gogol`, `koroche`, `tochkacuiri`, `listen`, `stories`,
`instruction`, `jazz_dance`) тот же класс переопределён **локально** под
принципиально другую вёрстку: `position:relative;min-height:100svh;
display:flex;padding:0`, а фон/видео рисуют соседние `.hero-bg`/
`.hero-overlay`/`<video>`. Если добавляешь новую страницу с видео-хиро —
бери за образец одну из этих, а не общий `.page-hero` из `assets/style.css`.

## Плавающая нижняя плашка и футер

На части страниц внизу есть фиксированный элемент — либо кнопки-CTA
(`.mob-cta`, показы/билеты), либо кнопка Telegram (`.tg-widget`, обычные
страницы). Оба перекрывают низ экрана на мобильных. Если у страницы есть
любой из них — у футера обязательно должен быть запас снизу
(`padding-bottom` в медиа-запросе `≤760px`, сейчас 120–150px), иначе
реквизиты и ссылки в футере окажутся под плашкой при докрутке до конца.

## Био-страницы — самодостаточны намеренно

`abdulova`, `moshnikov`, `bychkova`, `ilyina`, `sachkov`, `dyba`,
`rezidents` — сознательно **не участвуют** в общих CSS-переносах и держат
весь свой CSS локально, даже если он дублирует общий файл. Так решили
специально, чтобы правки на этих страницах не задевали остальной сайт и
наоборот. Если планируешь переносить их стили в `assets/style.css` —
сначала уточни, актуально ли ещё это решение.

(`res.cloudinary.com/kostyamoshnikov`, с готовыми трансформациями вроде

## Страницы под конкретную аудиторию (`business`, `event`, `creativity`, `venues`)

Четыре страницы — один и тот же шаблон: герой (заголовок + буллиты) → форма
с полями именно под эту аудиторию → тот же общий футер. `/collaboration`
собирает все пять направлений (включая «Партнёрам и инвесторам», которые
физически ведут на `/business`) в виде табов и просто ссылается на эти
страницы. Если добавляешь новую аудиторию — копируй один из этих файлов
целиком как основу, а не собирай с нуля.

У каждой формы — свой суффикс id (`_bz`, `_ev`, `_cr`, `_cl`, `_vn`) и своя
функция `handleForm(id)`, которая знает именно про её поля. Общий шаблон
формы (через `select`+один `textarea`) подходит не всем — если в форме
несколько текстовых полей с разным смыслом (как в `venues` — техника, даты,
пожелания), `handleForm` там переписан вручную под конкретные `id` полей,
копипаст общей версии потеряет данные.

## `display` у `.btn-outline` / `.btn-gold` — не удаляй молча

У этих классов бывает локальное переопределение вида `.btn-outline{display:inline}`
на нескольких страницах (`gogol`, `porfiriy`, `tickets`, `iov`) — оно там
специально, не мусор. Но **должно быть `inline-block`, не `inline`**: если
текст кнопки длинный и переносится на две строки (например, «Узнать о новых
датах»), у `inline`-элемента рамка не может аккуратно обернуть многострочный
текст — вылезают наползающие друг на друга линии. `inline-block` эту
проблему не имеет и всегда безопасен для кнопки с рамкой/паддингом.

## Плашки на `position:fixed` — считай реальную высоту шапки

Была страница (`/gogol`), где баннер «Архивный спектакль» стоял
`position:fixed;top:0` с `margin-top:60px`, чтобы отступить от шапки — но
шапка выше 60px, и баннер под неё залезал. Если элементу обязательно нужно
быть `fixed` поверх контента — не подбирай отступ на глаз под конкретный
брейкпоинт, высота шапки meняется (там сам логотип в две строки). Надёжнее
вообще не фиксировать такие второстепенные баннеры, а оставить в обычном
потоке документа сразу после шапки — так он гарантированно не наложится,
пусть и уедет при скролле вместе со страницей.

## Журнал чистки `assets/style.css`

Файл собирался в разное время, поэтому в нём накопился мусор. Чистка
ведётся аккуратно, по одной категории за раз, с проверкой баланса скобок
после каждого шага — не одним махом.

### Категория 1 — пустые комментарии-заголовки (сделано)

В файле было несколько мест вида:
```css
/* ── Cookie banner ── */



```
— заголовок-комментарий, под которым когда-то было CSS-правило, но правило
удалили, а сам заголовок и пустые строки после него — забыли. Визуально на
сайте это ни на что не влияло (браузер их просто не видит), но засоряло
файл и сбивало с толку при чтении: заголовок обещает код, а под ним пусто.

Удалено 5 таких блоков (заголовки: «Cookie banner» ×2, «Кнопка наверх»,
«МОБИЛЬНОЕ МЕНЮ» + «Оверлей за меню», «Telegram виджет») — итого 97 пустых
строк. **Ни одна строка с реальным CSS-правилом не тронута** — проверено
построчным сравнением с версией до правки: единственное отличие —
отсутствующие заголовки и пустые строки, весь код идентичен.
`693 → 596` строк.

### Категория 2 — дублирующиеся объявления (сделано)

Разбиралось по одному селектору за раз: правка → проверка баланса скобок →
проверка вычисленного эффективного стиля (до/после, программно) → следующий.
Ничего не удалялось массово без проверки.

**Итог — почищено:**

- `:root{--ink:...}` — дубль с идентичными значениями. Убран.
- `.logo{...}` — второе объявление было подмножеством первого (не хватало
  `color`/`text-decoration`, остальное совпадало). Убрано — эффект не изменился,
  недостающие свойства и так наследовались от первого правила.
- `@media(max-width:760px){.foot-grid{...}}` — сжатая копия существующего
  развёрнутого блока. Убрана.
- `@keyframes rise` — было два объявления с **разными** значениями (18px и
  16px). У `@keyframes` (в отличие от обычных правил) при совпадении имени
  побеждает целиком последнее по порядку объявление — так что 18px-версия
  была мёртвым кодом, реально анимация всегда шла с 16px. Убрано недостижимое.
- `a{color:inherit}`, `.logo small`, `.sec-head h2` — точные дубли или строгие
  подмножества. Убраны.
- `html{...}`, `body{max-width:100vw}` — по два-три объявления, часть друг
  друга полностью перекрывала (те же свойства, те же значения). Убраны лишние.
- `.sec-head::after` — было три объявления: одно без `content` (само по себе
  ничего не рисует), одно полное, одно — часть полного с теми же значениями.
  Проверено вручную, что итоговый вычисленный стиль одинаков что с одним
  полным правилом, что со всеми тремя. Оставлено одно полное.

**Сознательно не тронуто** (не дубли, а разные свойства одного селектора,
специально разбито на несколько правил — трогать не нужно):

- `footer a{color:var(--sand)}` — отдельной строкой под комментарием
  «Footer и nav — оставляем их собственные цвета», это фиксация решения.
- `nav` (position/z-index/background в одном правиле, padding — в другом)
- `.ph` (overflow/background в одном правиле, cursor — в другом)

`592 → 580` строк после этой категории (693 → 580 суммарно с категорией 1).
