# AELITA PRODUCTION — сайт

Статический сайт театральной продюсерской компании AELITA PRODUCTION.
Домен: **aelita-production.ru** (GitHub Pages, см. `CNAME`).

Чистый HTML/CSS/JS без сборки — что лежит в репозитории, то и деплоится.

## Структура

```
/
├── index.html              главная
├── <раздел>/index.html     каждая страница — своя папка + index.html
│                           (даёт чистые адреса вида /porfiriy, /tickets и т.д.)
├── assets/style.css        общие стили (см. ниже)
├── icons/, favicon.ico     иконки, manifest.json — PWA
├── sw.js, manifest.json    service worker и манифест для «Установить как приложение»
├── robots.txt, sitemap.xml
└── 404.html, offline.html  служебные страницы
```

## ⚠️ CSS: как устроено и что нельзя ломать

Стили лежат в двух местах:

1. **`assets/style.css`** — общее для всего сайта: шапка, мобильное меню, футер,
   кнопки, карточки событий, брейкпоинты. Подключается через
   `<link rel="stylesheet" href="/assets/style.css">`.
2. **`<style>` внутри каждой страницы** — то, что уникально именно для неё
   (свои отступы hero-блока, свой `--maxw`, специфичные компоненты).

**Три страницы НЕ подключают `assets/style.css` вообще** и должны быть
полностью самодостаточны — весь нужный CSS у них внутри `<style>`:

- `404.html`
- `offline.html`
- `app/index.html`

Это осознанно: страница 404 и офлайн-заглушка обязаны отрисоваться, даже
если с сайтом или сетью что-то не так. Если добавляешь этим трём страницам
новый визуальный элемент — стили для него нужно писать **прямо в их
собственном `<style>`**, не полагаясь на общий файл.

### Главное правило при правке общих стилей

Внешний `assets/style.css` в каскаде всегда идёт **раньше** локального
`<style>` страницы. Значит:

- Если у страницы есть **своё** базовое правило для селектора (например,
  свой `.sec-head{margin-bottom:...}`), а в `assets/style.css` для этого же
  селектора добавить `@media(...)`-переопределение — оно **не сработает**
  на этой странице, потому что локальное правило (без медиа-условия) всегда
  идёт позже и просто перебивает медиа-запрос.
- Если правишь что-то в `assets/style.css`, **проверяй, нет ли у страниц
  своего локального правила с тем же селектором** — если есть, адаптивное
  переопределение нужно продублировать и в `<style>` этой страницы тоже.
- В `assets/style.css` не должно быть **двух голых (без `@media`) правил
  для одного и того же селектора** — выигрывает то, что физически ниже по
  файлу, и это легко случайно поломать одной группе страниц в пользу другой.

Если что-то в вёрстке выглядит иначе на разных страницах непонятно почему —
в 9 случаях из 10 причина именно в этом.

### `:root{--maxw}`

Ширина контейнера контента — переменная, разная для двух групп страниц:
- `1320px` — все обычные страницы (задано в `assets/style.css` по умолчанию)
- `900px` — юридические страницы (`oferta`, `privacy`, `cookies`,
  `documents`, `bot`), заданы локально в каждой из них

## Юридические страницы и индексация

`robots.txt` закрывает от индексации `/oferta`, `/privacy`, `/cookies` —
это осознанно, не баг. При добавлении новой страницы не забудь внести её в
`sitemap.xml`.

## Дизайн-токены

```css
--ink:  #0B0B0D   фон
--coal: #1A1A1F   фон карточек/секций
--bone: #E8E6E1   основной текст
--sand: #C9B8A3   вторичный текст, подписи
--gold: #D6B57A   акцент, кнопки, hover
```

Шрифты — Montserrat (заголовки, лейблы) и Inter (текст), подключены через
Google Fonts (`fonts.googleapis.com`). Сайт не будет выглядеть как задумано
без доступа к интернету на этапе загрузки — если понадобится офлайн-шрифт,
нужно будет самостоятельно захостить `.woff2` локально.

## `.page-hero` — один класс, два разных смысла

На части страниц `.page-hero` — простой текстовый блок (заголовок + подпись,
статичный фон, обычный `padding`). На страницах с видео на весь экран
(`porfiriy`, `iov`, `gogol`, `koroche`, `tochkacuiri`, `listen`, `stories`,
`instruction`, `jazz_dance`) тот же класс переопределён **локально** под
принципиально другую вёрстку: `position:relative;min-height:100svh;
display:flex;padding:0`, а фон/видео рисуют соседние `.hero-bg`/
`.hero-overlay`/`<video>`. Если добавляешь новую страницу с видео-хиро —
бери за образец одну из этих, а не общий `.page-hero` из `assets/style.css`.

## Плавающая нижняя плашка и футер

На части страниц внизу есть фиксированный элемент — либо кнопки-CTA
(`.mob-cta`, показы/билеты), либо кнопка Telegram (`.tg-widget`, обычные
страницы). Оба перекрывают низ экрана на мобильных. Если у страницы есть
любой из них — у футера обязательно должен быть запас снизу
(`padding-bottom` в медиа-запросе `≤760px`, сейчас 120–150px), иначе
реквизиты и ссылки в футере окажутся под плашкой при докрутке до конца.

## Био-страницы — самодостаточны намеренно

`abdulova`, `moshnikov`, `bychkova`, `ilyina`, `sachkov`, `dyba`,
`rezidents` — сознательно **не участвуют** в общих CSS-переносах и держат
весь свой CSS локально, даже если он дублирует общий файл. Так решили
специально, чтобы правки на этих страницах не задевали остальной сайт и
наоборот. Если планируешь переносить их стили в `assets/style.css` —
сначала уточни, актуально ли ещё это решение.

## Изображения

Хостинг картинок не унифицирован: часть страниц использует Cloudinary
(`res.cloudinary.com/kostyamoshnikov`, с готовыми трансформациями вроде
`w_800,q_auto,f_auto`), часть — старый Tilda CDN (`thb.tildacdn.com`) без
какой-либо оптимизации по размеру, часть — сторонние домены напрямую
(`cuire-fest.ru/assets/...`). При добавлении новых изображений предпочтителен
Cloudinary с параметрами `w_...,q_auto,f_auto` — он один даёт реальную
экономию веса. `loading="lazy"` — на всех img кроме первого/hero-изображения
на странице.


Telegram-бот компании (`@aelita_production_bot`) обслуживается отдельным
Cloudflare Worker (`worker.js`, живёт вне этого репозитория). Страница
`/bot` на сайте — просто описание бота, с самим ботом не связана кодом.
